# Agent 文档工程参考

> 供维护 Agent 文档体系时参考。指导如何写好长期记忆（agent-memory）、CLAUDE.md/AGENTS.md 等入口指令、skill 和 reference。
> 只收录经多方独立验证、不随单个模型版本变化的通用原则。

---

## §1 上下文管理

1. **上下文越长，模型表现越差，且是连续下滑而非悬崖。** 对18个前沿模型的测试显示全部随输入增长而退化。实践中将有效上下文控制在窗口的25-30%时信噪比较高，但不应为凑这个数字而砍掉真正有用的信息。接 token 计数器告警，达阈值即触发压缩或开新会话。

2. **工具集要最小化。** 工具定义本身占用大量上下文——工具太多时，200k 窗口可能实际只剩70k 给真正的任务内容。经验法则：任意时刻启用不超过10个 server、活跃工具不超过80个。已有 CLI 能干活的场景，直接 shell out 比包成 MCP server 更省上下文。

3. **Subagent 要限定工具范围。** 5个工具的 subagent 比50个工具的更快更聚焦。子代理只返回摘要，搜索/试错/调试留在自己的上下文里，不回传到主上下文。

4. **Compact（压缩）策略需要显式配置。** 在文档中明确告诉 agent 压缩上下文时必须保留什么（API 变更及理由、错误信息及解法、已修改文件列表），只简要总结探索过程。不配置的后果是压缩时丢失关键决策记录。

5. **Cache hit rate 是生产 agent 系统的基础指标。** 80%是基准线，低于70%通常意味着 prompt 组装有 bug。三条提升规则：(1) 未用的 tool schema 不进上下文；(2) 原始 tool 输出不进上下文；(3) 用程序化方式让模型只取需要的部分。

6. **工具输出用 Markdown 而非 JSON。** 去掉无用字段和时间戳，改用 Markdown 格式，token 用量可降约66%且结果更好。这是成本最低的单点优化之一。

7. **静态数据放 prompt 开头。** 利用 context caching 机制，cache hit 价格约为 miss 的 1/10。不变的内容放前面，变化的内容放后面。

8. **一个会话只做一类任务。** 多种无关任务混在同一会话会互相污染上下文（"厨房水槽式会话"），上下文满了就开新会话。

## §2 文档内容取舍

1. **只写 agent 从代码库和 README 无法推断的信息。** 自检公式：对每一行问"如果删掉这行 agent 会出错吗？"——不会就删掉。架构概述段通常可以去掉，因为 agent 能从代码结构中推断出来。

2. **CLAUDE.md/AGENTS.md 超过约150行后回报递减。** 系统提示本身已占约50条指令位置，用户实际可用额度约150条。超过约200行时指令被"稀释"——遵从率悄悄下降，且没有任何报错提示。本质是尽可能精简，但如果内容确实都有用，硬砍行数反而有害。

3. **Commands 段（构建/测试/部署命令）是最高 ROI 组件。** 大规模仓库分析发现这个段持续贡献最大性能提升，应优先写好。

4. **代码片段优于文字描述。** 3-10行真实代码示例可提升代码复用率约20%。当需要展示规范用法时，给代码而非说教。

5. **每个 don't 配一个 do。**
   - 不好：`不要使用 class components`
   - 好：`使用 React hooks 而非 class components`
   
   底层原因：超过15个连续否定规则而无对应正面示范，会让 agent 过度保守、减少行动。

6. **可复用工作流放 Skills，不要放 CLAUDE.md。** CLAUDE.md 每次会话全量加载，Skills 按需加载。区分三种扩展机制的适用场景：
   - **常驻指令（CLAUDE.md）**：每次会话都需要的全局约束和偏好
   - **Commands**：用户主动输入斜杠才触发的偶尔操作
   - **Skills**：监听对话自动匹配触发的可复用工作流

7. **CLAUDE.md 的改动应当像代码改动一样被审查。** 它改变所有使用者的 agent 行为，影响面广。

8. **不要重复 linter 能做的事。** 格式规则让 linter 管，LLM 在这上面既慢又贵。

## §3 工作流与协作

1. **研究 → 规划 → 执行 → 审查是被多方独立验证的通用循环。** 对复杂、模糊或高影响任务，可以先用规划模式让 agent 出方案并校准方向；用户已明确授权且范围清楚时可直接执行。对复杂或模糊任务，可以让 agent 先访谈你来明确需求规格，然后基于规格文档执行——这个流程能发现大量"自己都没意识到自己不知道的问题"。

2. **Writer/Reviewer 分离消除确认偏误。** 一个 agent 会话写内容，另一个独立会话或子代理做审查。效果显著优于单 agent 自审——新鲜上下文的 Reviewer 不会对刚产出的内容产生确认偏误。

3. **Eval-driven development：先定评估标准再实现。** 每次改动用 eval 分数而非直觉来衡量。流程：先定义什么是好表现 → 用少量 eval 迭代到稳定 → 逐步扩大 eval 集 → 每个反馈都变成新 eval。本质是"自动化你能验证的，而非你能指定的"——有清晰验证标准，agent 才能自我迭代；没有验证标准的任务，agent 只能靠一次猜对。

4. **复杂任务先校准方案。** agent 越快，审查越重要。对复杂、模糊或高影响任务，先校准方案；用户已明确授权时直接推进，并展示证据（测试输出、命令返回），而非只口头断言"已完成"。把验证提前（shift verification left），而非事后补救。

5. **给 agent 可验证的目标。** 写测试、配 linter、在提示里加停止条件。"先写测试让 agent 能迭代"被多方独立验证为有效模式。

6. **对重复出现的错误才加规则。** 不要过早过度优化文档。只有某个错误反复出现时，才把纠正规则写入 CLAUDE.md。

## §4 已知陷阱

1. **LLM-as-Judge 有系统性偏差。** 用 agent 做自动评估时需注意：
   - 长度偏好：所有 LLM 评审系统性偏好更长的回答（约+17%），比人类评估的偏差更严重
   - 位置偏差：较弱的 judge 模型持续偏好第一个答案。缓解法：交换顺序跑两遍再平均
   - 自我增强：模型对自身生成的答案约+10%胜率

2. **Agent 默认不说"不知道"。** 模型不会主动承认不确定——它会自信地给出答案，包括错误答案。设计工作流时要假设 agent 不会主动暂停和澄清。如果需要 agent 在不确定时停下来问，必须显式在指令中说明何时应该暂停询问。

3. **LLM 自动生成的文档往往有害。** 在多数测试场景中降低成功率、增加步骤和成本。根本原因是自动生成的内容倾向于重复已有信息而非补充非显然信息。Agent 文档需要人的判断来决定什么该写什么不该写。

4. **超过15个连续否定规则的副作用。** 大量"不要做X"而没有对应的"做Y"，会让 agent 行为变得过度保守，宁可少做也不冒险——整体产出下降。

5. **指令被"稀释"没有报错提示。** 当 CLAUDE.md/AGENTS.md 内容超出有效额度时，遵从率悄悄下降，但不会有任何错误信息。这意味着你可能以为指令在起作用，但实际上 agent 已经开始忽略部分指令。唯一的检测方式是观察 agent 行为是否偏离预期。

6. **Hooks 是唯一"触发前不消耗上下文"的机制。** 它在错误发生瞬间注入修正，不预先填入上下文。如果某条规则只在特定错误出现时才需要，放 Hook 比放 CLAUDE.md 更合适。

7. **System prompt 要在"恰当的高度"。** 既不硬编码脆弱逻辑（太低层、容易因代码变动失效），也不过于含糊（太高层、agent 无法执行）。写倾向和原因而非硬约束——让 agent 理解为什么，而非只知道做什么。

## §6 外部参考：Claude Code AutoDream 记忆整理机制

来源：2026-03-31 Claude Code v2.1.88 npm source map 泄露（可还原源码）的二手分析（deeplearning.ai The Batch、claudefa.st、engineerscodex、VILA-Lab），非官方稳定文档；泄露时未正式发布或处于功能门控。可借鉴要点：

- **整理动作四分类**：合并重复、消除矛盾、未验证推测降级（resolve speculations）、清理过时。
- **相对时间绝对化**：沉淀长期记忆时把 yesterday/last week 类相对表述转为绝对日期。
- **权限级沙盒**：整理过程沙盒化，权限层面仅可写记忆目录，不能改源码/配置/测试——可作为无人值守审计"只出报告"约束的硬化方向。
- **窄搜索原则**：对可疑内容 grep 窄关键词而非穷尽读全文。

评估后可放弃的部分：四阶段命名（orient/gather/consolidate/prune）通常与既有审计流程等价，引入新术语违反"给启发式不给硬规则"；阶段阻塞预算适用于常驻 daemon，不适用于周期性批处理；空闲判定可由定时触发替代。
